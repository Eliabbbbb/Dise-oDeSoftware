Diseño de Software

Principios, mejores prácticas y fundamentos para crear software de calidad

1

¿Qué es el Diseño de Software?

Proceso de planificación de una solución informática que ayuda a los programadores a manejar la complejidad y disminuir el riesgo de errores.

Es donde se establece la calidad del proyecto y se materializan los requerimientos del cliente.

2

Decisiones Clave

  • Cómo se almacenarán los datos
  • Apariencia de interfaces de usuario
  • Comunicación con otros servicios
  • Delimitación de responsabilidades
3

Referencias IEEE

Requerimiento: Condición o capacidad que debe cumplir un sistema para satisfacer acuerdos o estándares.

Basado en IEEE 730-2014 e ISO/IEC/IEEE 24765 (2017).

IEEE 830:1998 establece características de buenas especificaciones: correctas, sin ambigüedades, completas, consistentes, etc.

Principios SOLID

SRP

Single Responsibility Principle: Un módulo debe tener una sola razón para cambiar.

OCP

Open/Closed Principle: Abierto para extensión, cerrado para modificación.

LSP

Liskov Substitution Principle: Los objetos deben ser reemplazables por sus subtipos.

ISP

Interface Segregation Principle: Interfaces específicas son mejores que generales.

DIP

Dependency Inversion Principle: Depender de abstracciones, no de implementaciones.

4

Otros Principios Importantes

  • DRY - Don't Repeat Yourself: Evitar duplicación de lógica
  • IoC - Inversion of Control: Delegar control de flujos
  • YAGNI - You Aren't Gonna Need It: No añadir funcionalidades prematuras
  • KISS - Keep It Simple, Stupid: Priorizar simplicidad
  • Ley de Demeter: Objetos no deben conocer estructuras internas de otros
5

Buenas Prácticas

  • Diseño débilmente acoplado
  • Composición sobre herencia
  • Encapsular lo que varía
  • Patrones de diseño
  • Regla del "Boy Scout": Mejorar código cuando se identifique oportunidad
  • Último Momento Responsable: Diferir decisiones irreversibles
6

Cuatro Reglas del Diseño Simple

  1. Los tests pasan
  2. Expresan intención (código autoexplicativo)
  3. No hay duplicidades (DRY)
  4. Mínimo número de elementos

Prácticas Técnicas Esenciales

+

Control de Versiones

Uso de Git, Subversion para gestionar cambios. Versionado de base de datos con herramientas como Flyway DB o Liquibase.

Pruebas

Unitarias: Verificar componentes individuales.

Integración: Probar componentes en conjunto con su entorno.

Calidad del Código

Código legible y comprensible. Establecer estándares de equipo. Usar herramientas como SonarQube.

Programación Segura

Conocer vulnerabilidades comunes. Seguir recomendaciones OWASP, SANS Institute en autenticación, validación, etc.

Documentación de Arquitectura

Es crucial documentar la estructura del software para futuras actualizaciones, mantenimiento o incorporación de nuevos desarrolladores.

Incluye diagramas de clases, actividades, secuencias y despliegue.

Estas prácticas proporcionan beneficios a largo plazo al construir sistemas robustos, flexibles y fáciles de mantener y escalar.
